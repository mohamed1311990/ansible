Ansible inventory: INI vs YAML

Same inventory, two formats — annotated side-by-side with a decision guide for when to use each.

hosts.ini INI format
# ungrouped hosts at the top bastion.example.com [webservers] web-01 ansible_host=10.0.1.10 web-02 ansible_host=10.0.1.11 ansible_port=2222 [databases] db-01 ansible_host=10.0.2.10 db-02 ansible_host=10.0.2.11 [webservers:vars] nginx_port=80 ansible_user=ubuntu [databases:vars] pg_version=16 ansible_user=ubuntu [production:children] webservers databases
[group] — section header defines a group
[group:vars] — variables for all hosts in the group
[group:children] — makes a group of groups
Host vars go inline: host key=value key=value
hosts.yml YAML format recommended
--- all: hosts: bastion.example.com: children: webservers: hosts: web-01: ansible_host: 10.0.1.10 web-02: ansible_host: 10.0.1.11 ansible_port: 2222 vars: nginx_port: 80 ansible_user: ubuntu databases: hosts: db-01: ansible_host: 10.0.2.10 db-02: ansible_host: 10.0.2.11 vars: pg_version: 16 ansible_user: ubuntu production: children: webservers: databases:
all: is the root — every host belongs to it
children: holds sub-groups (replaces :children)
vars: sits inside the group block (not a separate section)
Indentation = hierarchy — structure is immediately visible
Both files produce identical results when Ansible parses them. Verify with: ansible-inventory -i hosts.yml --graph
Nested groups — INI INI format
[atlanta_webservers] web-atl-01 ansible_host=10.1.1.10 web-atl-02 ansible_host=10.1.1.11 [london_webservers] web-lon-01 ansible_host=10.2.1.10 web-lon-02 ansible_host=10.2.1.11 # Group of groups — defined separately [webservers:children] atlanta_webservers london_webservers [atlanta_webservers:vars] aws_region=us-east-1 datacenter=atl [london_webservers:vars] aws_region=eu-west-2 datacenter=lon [webservers:vars] nginx_port=80 # Hard to see parent-child at a glance # :children and :vars sections scattered
Group relationships defined in separate :children sections — you must read the whole file to understand hierarchy
With many groups, :vars and :children sections get scattered throughout the file
Nested groups — YAML YAML format recommended
--- all: children: webservers: vars: nginx_port: 80 children: atlanta_webservers: vars: aws_region: us-east-1 datacenter: atl hosts: web-atl-01: ansible_host: 10.1.1.10 web-atl-02: ansible_host: 10.1.1.11 london_webservers: vars: aws_region: eu-west-2 datacenter: lon hosts: web-lon-01: ansible_host: 10.2.1.10 web-lon-02: ansible_host: 10.2.1.11
Hierarchy is visually obvious — parent wraps children, vars sit inside their group
No need to jump around — group, its vars, and its hosts are all co-located
Supports complex values natively: lists, dicts, booleans — impossible in INI
INI Use when...
+You're just getting started and want minimal syntax to learn
+Small, flat inventory — a handful of hosts, no nested groups
+Inheriting an existing INI inventory you don't want to rewrite
+Quick ad-hoc use: pasting a host list into a one-off file
-Avoid for complex nested groups — relationships get scattered
-Avoid when variable values are lists or dicts — INI can't express them
YAML Use when...
+Starting any new project — make YAML the default from day one
+You have nested group hierarchies (region → env → role)
+Variables include lists, dicts, or multi-line values
+Team collaboration — easier to review in pull requests
+Generating inventory programmatically — YAML is easier to template
-Indentation errors cause cryptic failures — use a YAML linter
INI limitationVariable types INI can't express
INI values are always strings. Writing my_list=one,two,three in INI gives Ansible the string "one,two,three" — not a list. YAML handles this natively.
INI — strings onlylimited types
# Looks like a list — it is NOT [webservers:vars] allowed_ips=10.0.0.1,10.0.0.2 # Ansible receives: "10.0.0.1,10.0.0.2" # Booleans are strings too debug_mode=true # Received as string "true", not boolean
YAML — native typesrich types
# Real YAML list vars: allowed_ips: - 10.0.0.1 - 10.0.0.2 # Ansible receives an actual list # Real boolean debug_mode: true # Received as boolean True
Debug tip: ansible-inventory -i hosts.yml --list dumps the fully parsed inventory as JSON — use it to verify your variables are the correct types (list vs string, boolean vs string).

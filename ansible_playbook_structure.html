What each section doeskey concepts
  • A playbook is a YAML list — each - at the top level is a play
  • hosts: maps this play to an inventory group or pattern — the play runs against every host in that group
  • gather_facts: true (default) runs the setup module first, collecting OS, IP, memory info as variables
  • vars: defines play-scoped variables — available to all tasks and roles in this play only
  • pre_tasks run before roles. roles run next. tasks run after roles. post_tasks run last — this order is guaranteed
  • Multiple plays in one file share nothing — different hosts, different vars, different privilege settings
  • Avoid mixing roles: and tasks: in the same play — use pre_tasks/post_tasks for setup/teardown and keep task logic in roles
The name: field on a play is not optional in practice — it appears in the output header and is the only way to identify which play is running. Always write descriptive names.
Task execution order within a play
  • 1. pre_tasks + their handlers
  • 2. roles (in list order) + their handlers
  • 3. tasks + their handlers
  • 4. post_tasks + their handlers
Real-world multi-play deploy playbookdeploy.yml
--- # Play 1: drain from load balancer - name: Remove hosts from load balancer hosts: webservers serial: 1 ← one host at a time gather_facts: false ← skip — saves 2-3s/host tasks: - name: Deregister from ELB community.aws.elb_instance: instance_id: "{{ ansible_ec2_instance_id }}" state: absent # Play 2: deploy new version - name: Deploy application hosts: webservers serial: 1 become: true vars: app_version: "{{ image_tag | default('latest') }}" roles: - deploy_app # Play 3: smoke test before re-adding - name: Verify deployment hosts: webservers serial: 1 gather_facts: false tasks: - name: Check health endpoint ansible.builtin.uri: url: http://localhost:8000/health status_code: 200 # Play 4: re-add to load balancer - name: Re-register with load balancer hosts: webservers serial: 1 gather_facts: false tasks: - name: Re-register in ELB community.aws.elb_instance: instance_id: "{{ ansible_ec2_instance_id }}" state: present
Why split into multiple plays?design rationale
  • Each play targets a different concern — drain, deploy, verify, re-add — making the playbook self-documenting
  • If play 3 (verify) fails, play 4 (re-add) never runs — the host stays out of the load balancer automatically
  • gather_facts: false on plays that don't need system info saves 2-3 seconds per host
  • serial: 1 means Ansible processes one host fully through ALL plays before moving to the next — true rolling deploy
  • Plays share the inventory but nothing else — each play starts fresh with its own variable scope
Use import_playbook to keep each logical section in its own file and compose them in site.yml. Cleaner than one giant multi-play file for complex workflows.

Become — how privilege escalation worksbecome
# Play level — all tasks run as root - name: Install packages hosts: all become: true ← enable for whole play become_method: sudo ← default, usually omit become_user: root ← default, usually omit tasks: - name: Install nginx ansible.builtin.package: name: nginx state: present # Task level — only this task escalates - name: Mixed privilege play hosts: all become: false ← no sudo by default tasks: - name: Read a config ← runs as ansible_user ansible.builtin.slurp: src: /home/ubuntu/.bashrc - name: Install a package ansible.builtin.package: name: htop become: true ← only this task uses sudo - name: Write app config ansible.builtin.template: src: app.conf.j2 dest: /etc/app/app.conf become: true become_user: appuser ← become a specific user
Sudoers config for Ansiblesystem setup
  • Ansible connects as ansible_user (e.g. ubuntu) then runs sudo to become root
  • Target machine must have sudo installed and the Ansible user must have sudo rights
  • For automation, configure NOPASSWD — otherwise Ansible hangs waiting for a password prompt
# /etc/sudoers.d/ansible # Allow ubuntu to sudo anything without password: ubuntu ALL=(ALL) NOPASSWD: ALL # More restrictive — only specific commands: ubuntu ALL=(ALL) NOPASSWD: /usr/bin/apt, \ /bin/systemctl, /usr/bin/pip3
Become method options
  • sudo — default, works everywhere
  • su — for systems without sudo (rare)
  • pbrun — PowerBroker (enterprise)
  • pfexec — Solaris / OpenBSD
  • doas — OpenBSD alternative to sudo
  • dzdo — CyberArk (enterprise PAM)
Set become_method and become_user in ansible.cfg rather than repeating them in every playbook. Only override per-play when you need something different.

parallelismHow forks and serial interact▾
  • forks (in ansible.cfg) controls how many hosts run one task simultaneously — default 5, set to 20+ for cloud
  • serial (in the play) controls how many hosts go through the entire play at once — used for rolling deploys
  • Without serial: Ansible runs each task across ALL hosts in parallel (up to forks limit), then moves to next task
  • With serial: 1: host 1 completes all tasks → host 2 starts — true rolling deploy, zero downtime
  • With serial: "25%": processes 25% of hosts at a time — balance between speed and safety
# serial accepts: integer, percentage, or list serial: 1 # one host at a time serial: 3 # three hosts at a time serial: "25%" # 25% of total hosts serial: [1, 3, 10] # canary: 1 first, then 3, then rest # Canary pattern — safest for production deploys: # Deploy to 1 host first, verify, then 3, then rest

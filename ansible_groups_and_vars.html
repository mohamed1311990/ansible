Ansible host groups, nested groups & group variables

How inventory groups work, how to nest them, how group_vars/ is structured, and how variables inherit through the hierarchy.

Flat group inventoryhosts.yml
--- all: # implicit root group hosts: # ungrouped hosts bastion: ansible_host: 10.0.0.1 children: # named groups live here webservers: hosts: web-01: ansible_host: 10.0.1.10 web-02: ansible_host: 10.0.1.11 vars: # group vars inline nginx_port: 80 ansible_user: ubuntu databases: hosts: db-01: ansible_host: 10.0.2.10 db-02: ansible_host: 10.0.2.11 vars: pg_version: "16" ansible_user: ubuntu
all: is always the root — implicit, always exists
bastion under all.hosts lands in the ungrouped group automatically
vars: inside a group applies to every host in that group
Host-level vars override group vars for that host only
What Ansible sees--graph output
# ansible-inventory --graph @all: |--@ungrouped: | |--bastion # no named group |--@webservers: | |--web-01 | |--web-02 |--@databases: | |--db-01 | |--db-02 # Check a specific host's resolved vars: # ansible-inventory --host web-01 { "ansible_host": "10.0.1.10", "ansible_user": "ubuntu", ← from group "nginx_port": 80 ← from group } # Run against a group: ansible webservers -m ping ansible databases -m shell -a "uptime"
ansible-inventory --graph is your primary debugging tool for inventory structure
ansible-inventory --host web-01 shows all resolved vars for one host
Target any group in a playbook with hosts: webservers or on CLI with -l webservers
Nested group hierarchyhosts.yml
--- all: children: # --- leaf groups (actual hosts) --- web_eu: hosts: web-lon-01: ansible_host: 10.1.1.10 web-lon-02: ansible_host: 10.1.1.11 vars: aws_region: eu-west-2 datacenter: lon web_us: hosts: web-nyc-01: ansible_host: 10.2.1.10 web-nyc-02: ansible_host: 10.2.1.11 vars: aws_region: us-east-1 datacenter: nyc # --- parent group (no direct hosts) --- webservers: vars: nginx_port: 80 # all web hosts get this ansible_user: ubuntu children: web_eu: # reference by name web_us: # --- environment group --- production: vars: env: production app_log_level: WARNING children: webservers: databases:
Parent groups contain children: which reference leaf group names — no hosts directly
Vars on a parent group flow down to all children — nginx_port: 80 reaches every web host regardless of region
A host can belong to multiple groups simultaneously — use this for cross-cutting concerns (role + region + env)
Resulting tree & patterns--graph output
# ansible-inventory --graph @all: |--@production: | |--@webservers: | | |--@web_eu: | | | |--web-lon-01 | | | |--web-lon-02 | | |--@web_us: | | | |--web-nyc-01 | | | |--web-nyc-02 | |--@databases: | | |--db-01 | | |--db-02 # Targeting patterns: # All web hosts everywhere: hosts: webservers # Only EU web hosts: hosts: web_eu # All production hosts: hosts: production # Intersection — web hosts in production: hosts: webservers:&production # All prod hosts EXCEPT databases: hosts: production:!databases
The colon : is the group operator — :& = intersection, :! = exclusion
Nesting means web-lon-01 is defined once but reachable via web_eu, webservers, or production
group_vars/ directory layoutfile structure
inventories/ production/ hosts.yml group_vars/ all/ # applies to EVERY host main.yml vault.yml # encrypted secrets webservers/ # applies to webservers group main.yml vault.yml databases/ # applies to databases group main.yml vault.yml web_eu.yml # single file also valid web_us.yml host_vars/ web-lon-01/ # per-host overrides main.yml db-01.yml # single file also valid
Directory name or file name (without .yml) must exactly match the group name
Use a directory (e.g. webservers/) to split vars and vault into separate files — Ansible loads all *.yml inside
Use a single file (e.g. web_eu.yml) for simple groups with few variables
host_vars/ follows the same pattern — directory or single file named after the hostname
What goes in each fileexamples
# group_vars/all/main.yml --- env: production app_domain: examsim.eu docker_registry: docker.io/mohamed1311990 ansible_user: ubuntu ansible_ssh_private_key_file: ~/.ssh/prod_key # Reference vault vars — never put raw secrets here db_password: "{{ vault_db_password }}" app_secret_key: "{{ vault_app_secret_key }}" # group_vars/all/vault.yml (ansible-vault encrypted) --- vault_db_password: super_secret vault_app_secret_key: django_secret_here # group_vars/webservers/main.yml --- nginx_port: 80 nginx_worker_processes: auto app_port: 8000 # group_vars/databases/main.yml --- pg_version: "16" pg_max_connections: 200 pg_shared_buffers: 256MB pg_data_dir: /var/lib/postgresql/16/main
Convention: prefix every vault variable with vault_, then alias it in main.yml — playbooks reference the alias and never need decryption to read code
group_vars/all/ is the right home for connection settings that apply everywhere

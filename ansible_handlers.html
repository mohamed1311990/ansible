Basic handler anatomynginx role
# roles/nginx/tasks/main.yml --- - name: Install nginx ansible.builtin.package: name: nginx state: present - name: Deploy nginx.conf ansible.builtin.template: src: nginx.conf.j2 dest: /etc/nginx/nginx.conf mode: "0644" notify: Reload nginx ← queues the handler - name: Deploy site config ansible.builtin.template: src: site.conf.j2 dest: /etc/nginx/sites-available/app.conf notify: Reload nginx ← queues the SAME handler again ← but it still only runs ONCE - name: Enable site ansible.builtin.file: src: /etc/nginx/sites-available/app.conf dest: /etc/nginx/sites-enabled/app.conf state: link notify: Reload nginx ← still queued only once total # roles/nginx/handlers/main.yml --- - name: Reload nginx ← name MUST match notify string ansible.builtin.service: name: nginx state: reloaded ← reload (no dropped connections) become: true - name: Restart nginx ← separate handler for hard restart ansible.builtin.service: name: nginx state: restarted become: true
The rules handlers followbehaviour
  • A handler is defined exactly like a task — same YAML structure, same modules — but lives in handlers/main.yml
  • A task notifies a handler by name using notify: — the name must match exactly (case-sensitive)
  • No matter how many tasks notify the same handler, it runs exactly once — Ansible deduplicates the queue
  • Handlers only run if the notifying task reported changed: true — if the config file was already correct, the reload never happens
  • Handlers run after ALL tasks in the current section complete — not inline where notify appears
  • If a play fails mid-way, handlers that were queued will NOT run — use --force-handlers or force_handlers: true to override
  • Handlers run in the order they are defined in handlers/main.yml, not in the order they were notified
Prefer reloaded over restarted for nginx, apache, and similar services — a reload applies config changes without dropping existing connections. Use restarted only when a full process restart is required.
notify — target by namestandard
# Standard: notify by handler name - name: Deploy app config ansible.builtin.template: src: app.conf.j2 dest: /etc/app/app.conf notify: Restart app # Notify multiple handlers from one task: - name: Update TLS certificate ansible.builtin.copy: src: cert.pem dest: /etc/ssl/cert.pem notify: - Reload nginx - Reload haproxy - Send alert to Slack # handlers/main.yml - name: Restart app ansible.builtin.systemd: name: myapp state: restarted daemon_reload: true ← reload systemd if unit changed - name: Reload nginx ansible.builtin.service: name: nginx state: reloaded - name: Send alert to Slack community.general.slack: token: "{{ slack_token }}" msg: "TLS cert updated on {{ inventory_hostname }}" channel: #ops-alerts
listen — topic-based notificationadvanced
# listen lets multiple handlers respond to one topic # tasks/main.yml - name: Update nginx main config ansible.builtin.template: src: nginx.conf.j2 dest: /etc/nginx/nginx.conf notify: nginx config changed ← notify a topic - name: Update nginx vhost ansible.builtin.template: src: vhost.conf.j2 dest: /etc/nginx/sites-available/app.conf notify: nginx config changed ← same topic # handlers/main.yml - name: Validate nginx config ansible.builtin.command: cmd: nginx -t listen: nginx config changed ← subscribes to topic become: true - name: Reload nginx ansible.builtin.service: name: nginx state: reloaded listen: nginx config changed ← also subscribes become: true
  • listen decouples tasks from handler names — tasks notify a semantic topic, handlers subscribe independently
  • Multiple handlers can listen to the same topic and all fire — ideal for validate → reload sequences
  • Handlers with listen run in the order they appear in handlers/main.yml — put validate before reload
When exactly do handlers run?timing
# Handlers flush after each section, not once at end Play: Configure webservers pre_tasks: ← runs, flushes handlers after task 1 → notify: Reload nginx (queued) task 2 → notify: Restart app (queued) ↓ pre_tasks handler flush Reload nginx runs ✓ Restart app runs ✓ roles: ← runs, flushes handlers after role: nginx task 3 → notify: Reload nginx (queued) task 4 → no change (not queued) ↓ roles handler flush Reload nginx runs ✓ tasks: ← runs, flushes handlers after task 5 → notify: Reload nginx (queued) task 6 → notify: Reload nginx (queued, deduped) ↓ tasks handler flush Reload nginx runs ✓ (once, despite 2 notifies) post_tasks: ← runs, flushes handlers after
  • Handlers flush after each section: pre_tasks, roles, tasks, post_tasks — not once at the very end
  • Within each section, handlers run after ALL tasks in that section complete
  • A handler notified in tasks: runs before post_tasks: begin — letting you validate in post_tasks after the restart
Force handlers and meta flushcontrol
# Force handlers to run NOW (mid-play flush): - name: Deploy config ansible.builtin.template: src: app.conf.j2 dest: /etc/app/app.conf notify: Restart app # Flush handlers inline — runs Restart app NOW: - name: Flush handlers before health check ansible.builtin.meta: flush_handlers - name: Verify app is healthy after restart ansible.builtin.uri: url: http://localhost:8000/health status_code: 200 retries: 5 delay: 3 # force_handlers — run handlers even if play fails: - name: Configure web servers hosts: webservers force_handlers: true ← handlers run even on failure tasks: - name: Deploy config ansible.builtin.template: src: nginx.conf.j2 dest: /etc/nginx/nginx.conf notify: Reload nginx - name: This task fails ansible.builtin.command: false # Without force_handlers: Reload nginx is skipped # With force_handlers: Reload nginx still runs
  • meta: flush_handlers is the correct pattern when you need a service restarted before the next task — e.g. restart app, then immediately health-check it
  • force_handlers: true is useful for cleanup handlers that must run even when the play fails
patternHandler chaining — handlers that notify other handlers▾
# handlers/main.yml # Handlers can notify other handlers: - name: Validate nginx config ansible.builtin.command: cmd: nginx -t notify: Reload nginx ← on success, notify next handler become: true - name: Reload nginx ansible.builtin.service: name: nginx state: reloaded become: true # tasks/main.yml notifies only the first handler: - name: Deploy nginx config ansible.builtin.template: src: nginx.conf.j2 dest: /etc/nginx/nginx.conf notify: Validate nginx config # Chain: config deploys → validate runs → # if valid, reload runs # If validate fails, reload is never triggered
  • Handler chaining is the safe pattern for config validation — validate first, reload only on success, avoid reloading a broken config
  • Chained handler notifications are queued and run in the next flush cycle, not immediately
patternHandlers in roles vs playbooks▾
# Role handlers: roles/nginx/handlers/main.yml # Scoped to the role — not visible outside it # Playbook-level handlers live in the play itself: - name: Configure servers hosts: webservers handlers: ← play-level handlers - name: Restart nginx ansible.builtin.service: name: nginx state: restarted become: true tasks: - name: Deploy config ansible.builtin.template: src: nginx.conf.j2 dest: /etc/nginx/nginx.conf notify: Restart nginx # Scope rules: # - Role handlers: visible only within that role's tasks # - Play handlers: visible to all tasks in the play # - Name conflicts: play handler wins over role handler
  • Put handlers in the role that owns the service — the nginx role owns "Reload nginx", postgresql owns "Restart postgresql"
  • Play-level handlers are useful for cross-cutting actions spanning multiple roles — e.g. "Send deployment notification"
patternHandlers with delegate_to and run_once▾
# Handlers support all task keywords: when, delegate_to, etc. - name: Invalidate CDN cache ansible.builtin.uri: url: "https://api.cloudflare.com/client/v4/zones/{{ cf_zone }}/purge_cache" method: POST headers: Authorization: "Bearer {{ cf_token }}" body_format: json body: purge_everything: true delegate_to: localhost ← API call from control node run_once: true ← one API call, not one per host - name: Restart only on systemd hosts ansible.builtin.service: name: nginx state: restarted when: ansible_service_mgr == "systemd" become: true
  • delegate_to: localhost + run_once: true in a handler is the pattern for API calls that should happen once after any host changes — CDN invalidation, Slack notifications, webhook triggers
Common mistakesgotchas
  • Handler name must match notify string exactly — "reload nginx" and "Reload nginx" are different handlers. Ansible silently ignores a notify with no matching handler
  • Handlers only run when changed: true — if you force changed_when: true the handler fires every run; if you use changed_when: false the handler never fires
  • A failed play skips queued handlers — if task 5 fails, handlers queued by tasks 1-4 never run. Use force_handlers: true or meta: flush_handlers before the risky task
  • Handlers run in definition order, not notification order — if you need validate before reload, define validate first in handlers/main.yml
  • A handler notified across multiple hosts runs once per host — not once globally. Each host tracks its own queue independently

Does the notify string match handler name exactly? # grep -r "notify:" roles/ playbooks/ # grep -r "name:" roles/*/handlers/ # # 3. Is the handler in the right scope? # (role handler vs play handler)
The most common handler bug: a task uses changed_when: false for a good reason but also has a notify: that will never fire. Every task with both changed_when: false and notify: is always a bug — pick one.

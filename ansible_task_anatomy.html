The minimum viable taskessentials only
# Absolute minimum — just a name and module: - name: Install nginx ansible.builtin.package: name: nginx state: present # Most common real-world task shape: - name: Deploy nginx config ansible.builtin.template: src: nginx.conf.j2 dest: /etc/nginx/nginx.conf owner: root group: root mode: "0644" become: true notify: Reload nginx # FQCN — fully qualified collection name # Always use it — avoids ambiguity: ansible.builtin.copy: ← collection.namespace.module community.general.ufw: amazon.aws.ec2_instance: kubernetes.core.k8s:
  • name: — always write a descriptive sentence, not a label. "Install nginx" is good. "nginx" is not
  • Module name goes at the same indent level as name: — it IS the task type
  • Always use FQCN — makes the collection source explicit and avoids name conflicts between collections
  • A task with no name: shows the module and args in output — ugly and hard to read in CI logs
register — capturing task outputregister
# register stores the module's return value - name: Get service status ansible.builtin.command: cmd: systemctl is-active nginx register: nginx_status ignore_errors: true ← don't fail if not active # Use the registered var in the next task: - name: Start nginx if not running ansible.builtin.service: name: nginx state: started when: nginx_status.rc != 0 # register with shell — capture stdout: - name: Get current app version ansible.builtin.shell: cmd: cat /etc/app/VERSION register: app_version changed_when: false ← reading is never a change - name: Print current version ansible.builtin.debug: msg: "Running version: {{ app_version.stdout }}"
What a registered variable containsreturn values
# Every registered var has these standard keys: nginx_status: changed: false ← did the task change anything? failed: false ← did the task fail? rc: 0 ← return code (command/shell) stdout: "active" ← standard output stderr: "" ← standard error stdout_lines: ["active"] ← stdout split by newline # Module-specific keys (e.g. stat module): file_stat: stat: exists: true size: 1024 mode: "0644" isdir: false # Common access patterns: "{{ nginx_status.rc }}" ← return code "{{ nginx_status.stdout }}" ← full output "{{ nginx_status.stdout_lines }}" ← as list "{{ nginx_status.failed }}" ← boolean "{{ nginx_status.changed }}" ← boolean # Safe default if key might not exist: "{{ nginx_status.stdout | default('') }}"
  • Use debug: var: my_result to dump the full structure of any registered variable during development
  • Registered variables only exist for the current play — they don't persist to the next play
  • With loop:, the registered var gets a results key — a list of return values, one per iteration
when — conditionalswhen
# when uses Jinja2 expressions (no {{ }} needed) # OS family check — most common pattern: - name: Install on Debian systems ansible.builtin.apt: name: nginx when: ansible_os_family == "Debian" - name: Install on RedHat systems ansible.builtin.dnf: name: nginx when: ansible_os_family == "RedHat" # Multiple conditions — AND (list syntax): - name: Run only on prod Ubuntu 22 ansible.builtin.debug: msg: production ubuntu 22 when: - env == "production" - ansible_distribution == "Ubuntu" - ansible_distribution_version == "22.04" # OR condition: - name: Run on Ubuntu or Debian when: > ansible_distribution == "Ubuntu" or ansible_distribution == "Debian" # Based on registered result: - name: Reload if config changed ansible.builtin.service: name: nginx state: reloaded when: config_result.changed # Variable existence check: - name: Run only if var is defined when: my_var is defined - name: Run only if var is not empty when: my_var is defined and my_var | length > 0
failed_when and changed_whenoutcome control
# changed_when — override Ansible's change detection - name: Check if migration is needed ansible.builtin.command: cmd: python manage.py showmigrations register: migration_status changed_when: false ← reading never changes anything - name: Run migrations ansible.builtin.command: cmd: python manage.py migrate register: migrate_result changed_when: "'No migrations to apply' not in migrate_result.stdout" # failed_when — custom failure condition: - name: Run test suite ansible.builtin.shell: cmd: pytest tests/ --tb=short register: test_result failed_when: - test_result.rc != 0 - "'passed' not in test_result.stdout" # until / retries / delay — retry pattern: - name: Wait for app to be healthy ansible.builtin.uri: url: http://localhost:8000/health status_code: 200 register: health_check until: health_check.status == 200 retries: 12 delay: 5 ← up to 60s total wait
  • changed_when: false is essential for read-only command/shell tasks — without it Ansible always reports a change
  • until + retries + delay is the standard pattern for waiting on async processes — app startup, DNS propagation, cert issuance
loopBasic loop — iterate over a list▾
# loop: replaces the deprecated with_items: - name: Install required packages ansible.builtin.package: name: "{{ item }}" ← item = current iteration state: present loop: - nginx - git - curl - htop # Loop over a variable (list from inventory/vars): - name: Create application directories ansible.builtin.file: path: "{{ item }}" state: directory mode: "0755" loop: "{{ app_directories }}" ← defined in vars become: true
  • The current iteration value is always available as item inside the task
  • Avoid using loop: with the package module for large lists — pass the list directly to name: instead (one module call vs N calls)
loopLoop over dicts — multiple values per iteration▾
# Loop over a list of dicts: - name: Create system users ansible.builtin.user: name: "{{ item.name }}" groups: "{{ item.groups }}" shell: "{{ item.shell | default('/bin/bash') }}" state: present loop: - name: appuser groups: docker shell: /bin/bash - name: deployer groups: sudo,docker - name: monitor groups: adm shell: /bin/false become: true # loop_control — customise loop behaviour: - name: Deploy vhost configs ansible.builtin.template: src: vhost.conf.j2 dest: "/etc/nginx/sites-available/{{ item.name }}.conf" loop: "{{ nginx_vhosts }}" loop_control: label: "{{ item.name }}" ← cleaner output (not full dict) loop_var: vhost ← rename item → vhost pause: 1 ← seconds between iterations
  • loop_control.label is essential when looping over dicts — without it, the full dict prints for every iteration
  • loop_control.loop_var renames item to something meaningful — critical when nesting loops in included tasks
looploop with register — capturing results per iteration▾
# When using loop + register, results is a list: - name: Check each service ansible.builtin.command: cmd: "systemctl is-active {{ item }}" loop: [nginx, postgresql, redis] register: service_checks ignore_errors: true changed_when: false # service_checks.results is a list, one item per loop: - name: Show failed services ansible.builtin.debug: msg: "{{ item.item }} is NOT running" loop: "{{ service_checks.results }}" when: item.rc != 0 loop_control: label: "{{ item.item }}" ← item.item = original loop value
  • With loop + register, the registered var has a results key — a list of return dicts, one per iteration
  • Each entry in results has an extra item key containing the original loop value — use item.item to access it

Common mistakes and fixesgotchas
  • command and shell always report changed: true — always add changed_when: false for read-only commands
  • when: uses Jinja2 expressions WITHOUT {{ }} — writing when: "{{ my_var == 'yes' }}" is deprecated; write when: my_var == 'yes'
  • Looping with package one item per iteration is slow — pass a list directly: name: [nginx, git, curl]
  • Registered variables from a loop have a results key — not a direct return value. Access my_result.results[0].stdout, not my_result.stdout
  • Use ansible.builtin.debug: var: my_result to dump any registered variable and see its full structure
  • Use no_log: true on any task that handles passwords, API keys, or tokens
  • ignore_errors: true continues on failure and tracks it. failed_when: false marks the task as always succeeded. They are different — use the right one for your situation
